# V7 → V8 → V9 重建升級路徑：實作清單與紀錄（新主線）

> 分支：feat/v7-clean-auth-store
> 原則：先打穩新 V7 的分層與型別地基，再往新 V8、新 V9 升級。

---

## 已確認的設計決策

1. `userId` 從新 V7 起固定為 `string`。
2. `userId` 視為 opaque string；`"0001"` 只是示例格式。
3. `Auth` 與 `Store` 分離：
   - `Auth`：登入與使用者身分查找
   - `Store`：menu/orders/order_items 業務資料存取
4. `Route` 只協調 `Auth` 與 `Store`，不混入底層細節。

---

## 這一步的背後邏輯（新 V7 為什麼要先做這件事）

新 V7 不是單純把舊程式碼改一點點，而是在重建一條之後可以一路升到新 V9 的乾淨主線。

這一步先做 `Auth` / `Store` 分離，有三個原因：

1. 避免責任混層

- 舊版本把 `login()`、`getUserById()` 這類身份責任放在 `Store`
- 這會讓「資料存取」與「認證機制」糾纏在一起
- 一旦未來更換登入方式，Store 就會被迫一起改

2. 把可變的東西隔離起來

- 未來 V7 → V8 會從 JsonFile 進到 DB
- V8 → V9 會從 DemoAuth 進到 Better Auth
- 若 `Auth` 與 `Store` 先分離，後面每次升級只需要替換其中一層，不用整條重寫

3. 把 `userId` 型別穩定下來

- 如果新 V7 還沿用 number，到了新 V9 又會重演一次型別翻轉
- 因此新主線從一開始就固定 `userId: string`
- 之後就算從 `"0001"` 換成 UUID，也只是值格式改變，不是型別改變

一句話總結：

> 新 V7 先解決的是「架構邊界」與「型別穩定性」，不是先追求更多功能。

---

## 這一步的實作順序（為什麼照這個順序做）

這一輪的順序不能亂，因為每一步都依賴前一步的結果。

1. 先改契約與型別

- `shared/contracts.ts`
- 先把 `User.id`、`Order.userId` 固定成 `string`
- 這是最上游的事實

2. 再改 `Store` 介面

- `store/Store.ts`
- 先把 `login()`、`getUserById()` 從介面移除
- 再把所有 `userId` 參數改成 `string`

3. 接著補 `Auth` 抽象

- `auth/Auth.ts`、`auth/DemoAuth.ts`、`auth/index.ts`
- 讓原本屬於 Store 的登入責任有地方可去

4. 再改 `Store` 的具體實作

- `store/json/JsonFileStore.ts`
- 把 auth 方法刪掉，保留業務資料存取
- 同時把舊資料中的數字 userId normalize 成字串

5. 最後才改 route

- `backend.ts`
- route 改為協調 `auth` 與 `store`
- 也就是：登入找 `auth`、訂單資料找 `store`

如果順序反過來，例如先改 route、但 `Auth` 還不存在，或 `Store` 介面還沒清乾淨，整體就會卡在半套狀態。

所以這一步的正確順序是：

```text
contracts → Store interface → Auth abstraction → Store implementation → route
```

## 實作紀錄 #002（運行環境收斂與故障排查）

- 日期：2026-04-23
- 背景：同時存在 `00_demo01` 與舊 worktree（如 `00_demo01_v7_clean`）執行紀錄，導致排查訊號混雜。

### 觀察到的症狀

1. `ENOENT: ./public/index.html`、`ENOENT: .../public`
2. 前端偶發「加入購物車失敗」，但 API 有時又看起來正常
3. 終端出現 `exit code 137/143`、`Terminated`

### 根因判斷

1. 不是單一程式邏輯錯誤，而是「執行環境混線」：

- 多個 backend 行程同時存在
- 不同 cwd（含舊 worktree）都可能監聽 3000

2. `public/` 缺失屬於執行模式/目錄問題：

- 沒先 build frontend
- 或在非目標專案目錄啟動

### 收斂解法（標準流程）

1. 只保留單一來源目錄：固定使用 `00_demo01`
2. 清掉舊行程，避免 3000 被不同版本 backend 搶占：

```bash
fuser -k 3000/tcp || true
pkill -f "bun backend.ts" || true
pkill -f "bun run dev" || true
```

3. 重新產生整合前端資產：

```bash
bun run build:frontend
```

4. 只啟動一個 backend：

```bash
bun --watch backend.ts
```

5. 以 smoke test 驗證而非只看瀏覽器表象：

- `GET /`
- `GET /health`
- `POST /api/auth/login`
- `GET /api/orders/current?userId=0001`
- `PATCH /api/orders/:id`

### 教學結論（要寫進課堂口訣）

1. `dev` 模式看 `5173`，整合模式看 `3000`（前提是先 build frontend）。
2. 看到 `ENOENT public/index.html` 先查「目錄與 build 狀態」，不要先懷疑業務邏輯。
3. 看到 API 時好時壞，先查「是否有多 backend 行程」再查程式碼。

## 實作紀錄 #003（舊 V8 namespace 改造）

- 日期：2026-04-23
- 背景：舊 V8 程式仍在，但資料庫結構已受舊 V9（Better Auth 半套）影響，直接執行舊 V8 會失配。

### 為什麼做

1. 不希望回滾現有資料庫主線。
2. 需要讓舊 V8 在課堂上可獨立運作與示範。
3. 需要讓多版本能同時存在、互不干擾。

### 做了什麼

1. 建立舊 V8 專用 PostgreSQL schema（預設 `v8_legacy`）。
2. 在該 schema 建立舊 V8 所需表：`users/menu_items/orders/order_items`。
3. 新增舊 V8 專用入口與 store：`backend.v8.ts` + `legacy/v8/store/PgStoreV8.ts`。
4. 保留舊 V8 的 `userId:number` 契約，不強行混入新主線 string 身分。
5. 新增 `bun run v8:db:setup` 與 `bun run dev:v8` 方便初始化與啟動。

### 怎麼做（執行流程）

1. `bun run v8:db:setup`
2. `PORT=3010 V8_DB_SCHEMA=v8_legacy bun run dev:v8`
3. 以 `/health`、`/api/auth/login`、`/api/orders/*` 做 smoke test

### 結果

1. 舊 V8 在 namespace 隔離下可恢復正常運作。
2. 不影響現有 V9/Better Auth 表結構。
3. 教材可同時保留「舊 V8」與「新主線」對照。

## 實作紀錄 #004（feat/v8-namespace：以新 branch 改造原生 V8 加入 namespace）

- 日期：2026-04-25
- 分支：`feat/v8-namespace`（從 `feat/v8-drizzle-neon` 切出）

### 為什麼切新 branch 而不直接改舊 V8

1. **保留「改造前」的對照點**：`feat/v8-drizzle-neon` 是沒有 namespace 的原始 V8，學生可以直接 diff 兩個 branch，清楚看到「只改 `db/schema.ts`，migration 就自動處理其餘」。

2. **符合本課程的分支策略**：`main` = 穩定里程碑，`feat/*` = 進行中改造。切新 branch 讓改造過程透明、可回滾、可做 PR review。

3. **改造步驟本身就是教材**：`feat/v8-drizzle-neon` → `feat/v8-namespace` 的 diff 就是一份完整的「最小改動示範」，讓學生看到 Drizzle + pgSchema 的威力：API 層零改動，只改 schema 定義。

### 做了什麼

1. `git checkout feat/v8-drizzle-neon && git checkout -b feat/v8-namespace`
2. 修改 `db/schema.ts`：加入 `pgSchema("breakfast")`，把所有 `pgTable` 改為 `appSchema.table`
3. `bun run db:generate`：產生 migration SQL（建立 schema + 搬移資料表）
4. `bun run db:migrate`：執行 migration
5. 啟動後端驗證 API 正常（所有 route 零改動）

### 核心驗證點

- `PgStore.ts` 不需要任何修改：它 import 的是 table 物件，Drizzle 自動在 SQL 加上 `breakfast.` prefix
- `db/client.ts` 不需要修改
- Elysia routes（`backend.ts`）不需要修改
- **唯一的修改點：`db/schema.ts`**

### 教學口訣

> 把資料表放進 namespace = 改 schema 定義，不是改程式邏輯。
> 這正是「single source of truth + 分層設計」在資料庫層的體現。
