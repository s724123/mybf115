## 5. 開發與執行模式的速查（避免常見混淆）

課堂上最常混淆的是「為什麼 `bun run dev` 後，`3000` 看不到前端頁面」。

請固定用這條規則判斷：

1. `bun run dev`：前端看 `5173`，後端 API 看 `3000`。
2. 想讓 `3000` 同時提供前端頁面：先 `bun run build:frontend` 產生 `public/`。
3. 若沒有 `public/index.html`，`3000` 仍可提供 API，但不保證有前端首頁。

建議助教在課堂 Demo 時，先口頭重複一次：

`dev 看 5173，整合看 3000（前提是已 build frontend）。`

## 6. 分支管理策略（Branch Strategy）

本專案採用一個心智負擔最小的分支策略：

### 核心原則

> `main` 永遠是完全測試通過、可信任的穩定狀態。

| 分支     | 語意                                   |
| -------- | -------------------------------------- |
| `main`   | 通過驗證的里程碑版本，可直接使用或部署 |
| `feat/*` | 進行中的開發分支，不保證穩定           |

### 工作流程

1. 從 `main` 切出 `feat/xxx` 開始開發
2. 在 feature branch 上反覆迭代、測試
3. 確認功能完整、型別正確、build 成功後，merge 回 `main`
4. `main` 指向「目前教到哪、測到哪」的最新穩定版

### 為什麼這樣做

- **學生 clone 後直接可用**：不需要猜「現在應該切哪個分支」
- **名字即語意**：`main` 這個名字就代表可信任，不需要額外說明
- **降低認知負荷**：只需要記一條規則，而不是記一套複雜的 branching model

### 教學情境下的補充說明

每一個 V 版本（V7、V8、V9）都代表一個教學里程碑。當某個版本的開發與驗證完成，feature branch merge 回 `main`，代表「這個版本的課已經上到這裡，而且是正確的」。

這個策略與 3-2 節「需求改變時，資料也要果斷跟著改」的精神一致：**讓系統的狀態真實反映當前事實**，而不是讓歷史狀態繼續混淆現在。
